如果你只用过一种方式调用 AI——比如在 terminal 里敲 `claude`，或者在网页上跟 ChatGPT 对话——Multica 的四种触发方式可能会让你觉得有点多。但一旦理解每种方式对应的场景，你会发现它们不是在堆功能，而是在把"让 AI 接手"这个动作嵌入到已有的协作习惯里。

**第一种：分配 issue。** 这是最正式的方式。你创建一个 issue，写好需求，用 `multica issue assign` 或者 Web 界面分配给一个 agent。agent 会完整跟踪这个任务——从入队到完成，所有中间状态和输出都留在 issue 的时间线里。适合：有明确交付边界的任务、需要审计追踪的工作、多人协作的复杂需求。

**第二种：在评论里 @agent。** 你在一个已有的 issue 讨论中，突然想对 agent 说"你帮我看一下这段"。不需要创建新 issue，直接在评论里写 `[@agent名](mention://agent/<id>)`，平台会把这条评论当作一个新任务来触发。agent 能看到整个讨论上下文，于是在对话流中直接插入结果。适合：临时的代码审查请求、clarification 追问、在已有讨论中快速插入一个 AI 执行步骤。

**第三种：直接聊天。** Multica 提供了 agent 聊天界面，你可以像跟同事聊天一样直接对 agent 说话。没有 issue 的仪式感，没有状态流转——就是发消息、得回复。适合：草拟还没成型的想法、快速问答、"帮我查一下这个 API 的文档"这类无需正式追踪的请求。聊完之后如果觉得有价值，可以随时把聊天内容转化为一个正式 issue。

**第四种：Autopilots。** 这是定时或事件触发的自动化。你可以配置一个 autopilot，让它每天早上 9 点自动跑一次代码质量检查，或者每当有新的 issue 被标记为 `bug` 时自动触发一个诊断 agent。Autopilot 的本质是把"人主动分配"这个动作变成了"按规则自动触发"。适合：周报生成、每日巡检、内容沉淀等周期性任务。

这四种方式对应了四种不同的"触发力度的颗粒度"。正式任务用 issue，临时插话用评论 @，轻量探索用聊天，规律性的固定任务用 autopilot。

你可能会问：什么时候用哪种？一个简单的决策框架：

- 这件事需要**被追踪和回溯**吗？→ 用 issue。
- 这件事是**在已有讨论中冒出来的**吗？→ 用评论 @。
- 这件事还在**模糊探索阶段**吗？→ 用聊天。
- 这件事**周期性重复**且不需要每次都手动触发吗？→ 用 autopilot。

在实践中，我们发现最有价值的组合是：聊天探索 → 转成 issue 正式执行 → 执行过程中评论 @ 追加细节 → 成熟的固定模式沉淀为 autopilot。这是一个从模糊到精确、从一次性到自动化的自然演化路径。

不必一上来就全部掌握。从"分配 issue"开始——这是 Multica 最核心的交互模式，其他三种都是它的变体和补充。
