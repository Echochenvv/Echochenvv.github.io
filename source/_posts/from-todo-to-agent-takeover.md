几年前我们团队从 Jira 迁到 Linear 的时候，觉得项目管理已经进化到头了——更快的 UI、更轻的交互、更好的快捷键。但有一个问题从来没有被解决：**issue 创建之后，还是得有人去看、去理解、去动手。**

Multica 做的事情，就是在这一环上换了答案。

你在 Multica 的 Web 界面上创建一个 issue，写好需求和上下文，然后把它分配给一个智能体。三秒之后，你本地机器上的 daemon 领取了这个任务，Claude Code（或者 Codex、Cursor——取决于你配置了哪个工具）开始在新的工作目录里读代码、改文件、跑测试。你没有再打开 IDE，没有再切 terminal，没有再手动 git clone。**任务已经从"待办项"变成了"正在进行中的 AI run"。**

这就是 Multica 和传统项目管理工具最根本的区别。Linear 和 Jira 的终点是一个被分配出去的 issue；Multica 的终点是一个**已经被 AI 接手的任务**。

这种转变在体感上是质变的。传统链路里，你写一个 issue → 等同事看到 → 同事理解需求 → 同事动手。每一步都有延迟、都有沟通成本。而 Multica 的链路是：你写一个 issue → 分配 → daemon 领取 → AI 工具开始执行。中间没有人类上下文切换的开销。

这不是说 AI 能替代所有人工步骤——复杂决策、架构评审、代码 review 仍然需要人。但"把任务从纸面落到执行"这一步，AI 可以零延迟接手。你写完 issue 的瞬间，工作已经开始了。

有一个细节值得注意：在 Multica 里，issue 不只是一个文字描述，它包含了**可执行上下文**——可以附带代码仓库链接、branch 引用、甚至具体的文件路径。这让 AI 不需要"理解"你的意图然后再去找代码，而是直接被投送到执行环境中。这种设计意味着，用户需要把 issue 写成**交付规格**而不是备忘录。"帮忙看看那个 bug"不如"修复 checkout 页面在 Safari 下金额显示为 NaN 的问题，相关文件在 `src/pages/checkout.tsx`"。

从心智模型上说，Multica 不是"有 AI 功能的项目管理工具"，而是 **AI Agent Workspace**——一个让人类和 AI agent 在同一空间里协作的工作台。issue 是协作的入口，不是追踪的终点。

如果你刚开始用 Multica，一个简单建议：下次创建 issue 时，假设读到它的人只有 30 秒就要开始动手写代码。写得越具体，AI 第一次交付的质量就越高。

这并不是说传统 issue tracker 没用了——它们适合管理人类团队的工作流。但如果你已经在用 AI 编程工具写代码，Multica 让这些工具从"你打开 terminal 手动调用的助手"变成了"在后台等待任务分发的 worker"。这个转变不大，但刚好够改变你安排一天工作的方式。
