为什么 AI 任务跑在你的机器上

有一个问题 Multica 的新用户几乎都会问：“为什么 agent 不能直接在你们的服务器上跑？”

简短回答是：因为你的代码、你的密钥、你已经配好的工具都在你自己的机器上。让 agent 上服务器跑——技术上做得到，但意味着你要把代码推给平台、把 API key 交给平台、把所有工具重新在云端配置一遍。这不仅是安全风险，也是效率损失。

Multica 做了一个有意识的设计决策：智能体不跑在平台服务器上，而是在你自己的机器上。 Server 负责协作——管 issue、管队列、推送通知。但它从来不需要访问你的代码仓库、你的 .env 文件、你的 SSH key。

这个决策的实际好处比你想象的多。

首先是安全边界。 使用 Multica 时，你的 Claude API key、GitHub token、npm 私有源认证——这些敏感凭证始终只存在于你的本地机器上。Agent 执行需要的任何授权，用的都是你本机已经登录好的 session。平台服务器不需要、也不会接触到这些凭据。相比”把代码和密钥上传到某个云端 agent 服务”的模式，这种本地执行天然隔离了凭据泄露半径。

其次是工具复用。 你已经在本地配置了一套 AI 编程工具。Claude Code 你用惯了，Cursor 你买了 license，Codex 你配好了自定义 rules。Multica 的 daemon 会自动探测这些工具，并在任务到来时选择可用的来执行。不需要在平台侧重新搭建执行环境、重新写配置、重新调试工具链。”我本机能跑什么，Multica agent 就能用什么”——这是本地执行带来的零迁移成本。

第三是执行环境的可控性。 你的代码依赖特定版本的 Node、需要访问内网的私有 npm registry、要用本地安装的字体渲染图片——这些在本地都是”本来就正常”的事情。但如果 agent 在云端执行，每个环境差异都可能变成一个需要排查的问题。本地执行意味着 agent 的任务环境和你的开发环境是同一套，没有”在我机器上能跑”的 gap。

当然，这个设计有它的 trade-off。daemon 需要在线——你的笔记本合上了，agent 就没法领取新任务。本地工具需要可用——Claude Code 恰好 crash 了、token 过期了，任务就会 failed。机器性能也会影响执行效率——大型 repo 的 clone 和索引在你本机完成，SSD 速度和网络带宽都是变量。

但 Multica 的选择是务实的：与其在云端维护一套”和用户本地类似但永远不完全一致”的执行环境，不如把调度和协作放在云端，把执行留在本地。这个取舍的代价是用户需要维护一个健康的本地 daemon 状态；收益是安全、可控、零配置迁移。

把本地 daemon 当成你的 AI 执行网关——它就像你机器上的一个常驻 agent runner。它不写代码（那是 Claude Code 的工作），但它确保任务来了有人接、接了能跑、跑了有结果。
